# การใช้ทรัพยากรเมื่อดึง 20 จอขึ้นเว็บ (หลังสมัครเสี่ยวเหว๋ยแล้ว)

เอกสารนี้ตอบคำถาม: **เมื่อดึงหน้าจอ 20 เครื่องขึ้นแสดงบนเว็บ จะใช้ทรัพยากรของคอมพิวเตอร์เยอะไหม?**

---

## สรุปสั้นๆ (ตอบลูกค้า)

- **เครื่อง Server BoxPhone (ที่รันเสี่ยวเหว๋ย + ต่อ USB 20 เครื่อง)**  
  → ใช้ทรัพยากร **ปานกลาง** (CPU/RAM มีการทำงานทุก 5 วินาทีเมื่อมีการ refresh ภาพ 20 จอ)  
  → เครื่อง server ทั่วไป (เช่น CPU 4 core, RAM 8GB+) **รับไหว**

- **คอมของคนเปิดเว็บ Admin (ลูกค้า)**  
  → ใช้ทรัพยากร **น้อย** แค่รับภาพ 20 ภาพมาหน้าเว็บ แสดงใน grid  
  → คอมหรือโน้ตบุคทั่วไป **ไม่หนัก**

- **เครือข่าย**  
  → ประมาณ **1–2 MB ต่อการ refresh หนึ่งรอบ** (20 ภาพ)  
  → ถ้า refresh ทุก 5 วินาที ≈ **ประมาณ 0.2–0.4 MB/s**  
  → ใช้สาย LAN หรือ Wi-Fi ปกติ **เพียงพอ**

---

## ความเป็นจริงของระบบ (Technical)

### โครงสร้างการทำงาน

1. **เว็บ Admin** แสดง 20 การ์ด แต่ละการ์ด = 1 หน้าจอจาก 1 เครื่อง
2. แต่ละการ์ดจะ **ขอภาพหน้าจอ** จาก Backend (NestJS)  
   → ตอนโหลดหน้า + **รีเฟรชอัตโนมัติทุก 5 วินาที**
3. **Backend** ไปขอภาพจาก **เสี่ยวเหว๋ย** (WebSocket หรือ HTTP)  
   → เสี่ยวเหว๋ยดึงภาพจาก Android แต่ละเครื่อง (ผ่าน USB/ADB)
4. ภาพถูกส่งกลับไปที่เว็บ → แสดงใน grid

ดังนั้น:

- **ที่เครื่อง Server BoxPhone**
  - งานหลัก: **เสี่ยวเหว๋ย** capture หน้าจอ 20 เครื่อง (และส่งให้ Backend)
  - ทุก 5 วินาที = 20 ครั้ง capture ต่อรอบ
  - CPU: มีการใช้งานช่วงที่ capture (ขึ้นกับว่าเสี่ยวเหว๋ยทำแบบ parallel หรือทีละตัว)
  - RAM: ต้องเก็บ buffer ภาพ 20 ภาพชั่วคราว (ภาพละประมาณ 200KB–1MB ขึ้นกับความละเอียดและรูปแบบ)

- **ที่ Backend (NestJS)**  
  - ถ้ารันบนเครื่องเดียวกับเสี่ยวเหว๋ย: แค่รับคำขอจากเว็บ แล้วไปขอภาพจากเสี่ยวเหว๋ย แล้วส่งต่อ  
  → ใช้ CPU/RAM น้อย (เป็นแค่ตัวกลาง)

- **ที่คอมคนดูเว็บ**  
  - แค่รับ 20 ภาพ (JPEG/PNG) มาแสดง  
  → โหลดไม่หนักสำหรับเบราว์เซอร์สมัยใหม่

## ตัวเลขคร่าวๆ (ประมาณการ)

| จุดที่ใช้ทรัพยากร | ประมาณการ (20 จอ, รีเฟรชทุก 5 วินาที) |
|-------------------|----------------------------------------|
| **Server BoxPhone (CPU)** | ใช้งานช่วงสั้นๆ ทุก 5 วินาที ไม่ติด 100% ตลอด |
| **Server BoxPhone (RAM)** | เพิ่มขึ้นประมาณหลักสิบถึงหลักร้อย MB (buffer ภาพ) |
| **Network ต่อ 1 รอบ refresh** | ~2–6 MB (20 ภาพ × ประมาณ 100–300KB ต่อภาพ) |
| **คอมคนดูเว็บ** | ใช้ RAM/CPU น้อย แค่แสดง 20 ภาพในหน้าเว็บ |

## Real-time (รีเฟรชถี่ขึ้น) ได้ไหม?

**ได้** — ใช้โหมด **near real-time** โดยลดช่วงรีเฟรชจาก 5 วินาที เป็น **1 วินาที** (หรือ 1.5–2 วินาที)

### วิธีเปิดโหมด Real-time

ในโฟลเดอร์ **admin** สร้างหรือแก้ไข `.env.local` แล้วใส่:

```env
# รีเฟรชหน้าจอทุก 1 วินาที (real-time). ค่าเริ่มต้นไม่ใส่ = 5000 (5 วินาที)
NEXT_PUBLIC_SCREENSHOT_REFRESH_MS=1000
```

- **1000** = ประมาณ 1 วินาที (real-time)
- **2000** = 2 วินาที (ใกล้ real-time)
- **5000** = 5 วินาที (ค่าเริ่มต้น)
- ไม่ใส่หรือใส่ 0 = ใช้ 5 วินาที  
- ค่าต่ำสุดที่ระบบอนุญาต = **500 ms** (0.5 วินาที) — ใช้เฉพาะถ้าเครื่อง Server แรงมาก

จากนั้น **restart dev server ของ Admin** (หรือ build ใหม่) แล้วเปิดหน้า Overview / Devices ใหม่ หน้าจอจะอัปเดตตามช่วงที่ตั้งไว้

### ข้อควรระวัง (Real-time = โหลดมากขึ้น)

| โหมด        | ช่วงรีเฟรช | จำนวน request ต่อนาที (20 จอ) | ภาระ Server |
|------------|-------------|--------------------------------|-------------|
| ปกติ       | 5 วินาที    | 20 × 12 = 240                  | ปานกลาง     |
| Real-time  | 1 วินาที    | 20 × 60 = 1,200                | สูงขึ้น      |
| เร็วมาก    | 0.5 วินาที  | 20 × 120 = 2,400               | สูงมาก      |

ถ้าใช้ real-time (1 วินาที) กับ 20 จอ เครื่อง Server และเครือข่ายจะทำงานหนักขึ้นประมาณ **ประมาณ 5 เท่า** ของโหมด 5 วินาที — แนะนำให้ใช้กับเครื่องสเปกดี (เช่น 4 core ขึ้นไป, RAM 8GB ขึ้นไป)

**หมายเหตุ:** ระบบปัจจุบันดึงภาพแบบ **screenshot ทีละภาพ** (request–response) ไม่ใช่ video stream จริง ดังนั้น "real-time" หมายถึง **รีเฟรชถี่ขึ้น** (เช่น 1 ภาพต่อวินาที) ไม่ใช่ live stream แบบต่อเนื่องแบบวิดีโอ

## คำแนะนำถ้าไม่อยากให้เครื่อง Server หนักเกินไป

1. **ลดความถี่การรีเฟรช**  
   - ค่าเริ่มต้น **ทุก 5 วินาที**  
   - ถ้าเปลี่ยนเป็นทุก 10 หรือ 15 วินาที จะลดการ capture และการส่งภาพลงครึ่งหรือหนึ่งในสาม  
   - หรือใช้ **real-time** โดยตั้ง `NEXT_PUBLIC_SCREENSHOT_REFRESH_MS=1000` (ดูหัวข้อ Real-time ด้านบน)

2. **ปิด auto-refresh ได้**  
   - ในโค้ดมีปุ่ม "รีเฟรช" ให้กดดูเมื่อต้องการ  
   - สามารถปิดการรีเฟรชอัตโนมัติ (แก้ค่า interval ใน frontend) ได้ถ้าต้องการ

3. **เครื่อง Server แนะนำ**  
   - CPU 4 core ขึ้นไป, RAM 8GB ขึ้นไป  
   - ใช้ SSD จะช่วยเรื่อง I/O ตอนอ่าน/เขียน buffer ภาพ

## สรุปสำหรับลูกค้า (ข้อความที่ส่งได้เลย)

> **ในกรณีที่สมัครเสี่ยวเหว๋ยแล้ว เมื่อดึง 20 จอขึ้นเว็บ:**  
> - **เครื่อง Server BoxPhone** จะใช้ทรัพยากรปานกลาง (CPU/RAM มีการทำงานทุก 5 วินาทีตามการรีเฟรชภาพ) เครื่องสเปกทั่วไปอย่าง 4 core / 8GB RAM รับไหว  
> - **คอมของท่าน (ที่เปิดเว็บ Admin)** ใช้ทรัพยากรน้อย แค่รับและแสดง 20 ภาพ  
> - **เครือข่าย** ประมาณ 1–2 MB ต่อการรีเฟรชหนึ่งรอบ (ทุก 5 วินาที) ใช้สาย LAN หรือ Wi-Fi ปกติเพียงพอ  
>  
> ถ้าต้องการลดภาระที่ Server สามารถตั้งให้รีเฟรชหน้าจอช้าลง (เช่น ทุก 10–15 วินาที) หรือปิดการรีเฟรชอัตโนมัติ แล้วกดรีเฟรชเมื่อต้องการดูได้

## คำถามเพิ่ม: เครื่องสเปกนี้ไหวประมาณกี่ชุด?

**สเปกที่ลูกค้าให้มา:**  
- RAM: 16GB × 4 = **64GB** DDR4 bus 2400  
- CPU: **36 cores / 72 threads**, 2.3–3.0 GHz  

**สมมติ 1 ชุด = 1 BoxPhone set = 20 เครื่อง (ดึง 20 จอขึ้นเว็บ + งานอื่นในระบบ)**

### ประมาณการจาก CPU และ RAM

| ทรัพยากร | ใช้ต่อ 1 ชุด (ประมาณการ) | จากสเปกนี้ → ประมาณกี่ชุด |
|----------|---------------------------|----------------------------|
| **RAM**  | OS + แอป ~6–8 GB, ต่อชุด ~1–2 GB | 64 − 8 = 56 GB สำหรับชุด → **ประมาณ 25–35 ชุด** (ถ้าขึ้นกับ RAM อย่างเดียว) |
| **CPU**  | ต่อชุดใช้แบบ burst ทุก 5 วินาที ประมาณ 1–2 core เทียบเท่า (เฉลี่ย) | 36 cores → **ประมาณ 18–25 ชุด** (ถ้าขึ้นกับ CPU อย่างเดียว) |

### สรุปจำนวนชุด (จาก CPU + RAM)

- **แบบ conservative (ไม่ให้เครื่องเต็ม)**  
  → **ประมาณ 15–20 ชุด** (ชุดละ 20 เครื่อง = รวม 300–400 เครื่อง)
- **แบบกลางๆ**  
  → **ประมาณ 20–25 ชุด** (400–500 เครื่อง)
- **แบบดึงเต็มสเปก (และไม่มีจุดคอขวดอื่น)**  
  → **ไม่เกินประมาณ 25–30 ชุด** (500–600 เครื่อง)

### ข้อแม้สำคัญ (จุดคอขวดอื่นที่อาจมาก่อน CPU/RAM)

1. **USB / ฮาร์ดแวร์**  
   แต่ละชุดมักต่อผ่าน USB hub หรือการ์ดหนึ่งจุด ถ้าเครื่องนี้มี USB controller / พอร์ตจำกัด จำนวนชุดจริงจะ **ไม่เกินจำนวนจุดที่ต่อได้** (เช่น 4–8 ชุด ถ้ามี 4–8 hub ชุดละ 20 เครื่อง)

2. **ซอฟต์แวร์เสี่ยวเหว๋ย**  
   อาจจำกัดจำนวนอุปกรณ์ต่อ instance หรือต่อเครื่อง (ต้องดูจาก manual / ใบอนุญาต)

3. **เครือข่าย**  
   หลายชุด = หลายคนเปิดเว็บ Admin พร้อมกัน แบนด์วิธและจำนวน connection จะเพิ่มตาม

### ข้อความสรุปสำหรับส่งลูกค้า

> จากสเปก **RAM 64GB + CPU 36c/72t** ถ้าดูจาก CPU และ RAM อย่างเดียว เครื่องนี้รับโหลดแบบ "ดึง 20 จอขึ้นเว็บ" ได้ประมาณ **15–25 ชุด** (ชุดละ 20 เครื่อง) โดยแบบปลอดภัยแนะนำ **ประมาณ 15–20 ชุด**  
>  
> จำนวนชุดจริงอาจน้อยกว่านั้น ถ้า:
> - ต่อ USB ได้จำกัด (กี่ hub / กี่จุดต่อ 20 เครื่อง)
> - เสี่ยวเหว๋ยจำกัดจำนวนเครื่องต่อ instance  
>  
> แนะนำให้ทดสอบจริงที่ 1–2 ชุดก่อน แล้วค่อยเพิ่มชุดแล้วตรวจสอบการใช้ CPU/RAM และ USB
